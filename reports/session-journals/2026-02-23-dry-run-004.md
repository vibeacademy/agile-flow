# Session Journal — 2026-02-23 (Dry Run 004)

## Session Summary

Ran the `/doctor` health check on a fresh dry-run-004 instance and discovered that `scripts/doctor.sh` produces false positives for CLI tool checks (`gh`, `claude`, `npx`, `node`, `uv`) when invoked from Claude Code's sandboxed environment. The sandbox restricts `PATH` to `/usr/bin:/bin:/usr/sbin:/sbin`, causing `command -v` to miss binaries installed via Homebrew (`/opt/homebrew/bin`) or Claude's installer (`~/.claude/local`). This is a trust-eroding bug — participants who have correctly installed their tools will see FAILs and WARNs that aren't real, reducing confidence in the diagnostic and creating unnecessary troubleshooting work.

## Challenges and Mitigations

### 1. `gh` reported as FAIL despite being installed

| Field | Value |
|-------|-------|
| Challenge | `/doctor` reported `[FAIL] CLI Tools: gh not found` even though `gh` is installed and invocable from any terminal |
| Root cause | Claude Code runs Bash commands in a sandboxed environment with a minimal `PATH` (`/usr/bin:/bin:/usr/sbin:/sbin`). The user's shell profile (`.zshrc`) adds `/opt/homebrew/bin` to `PATH`, but the sandbox doesn't source it. `command -v gh` fails because `/opt/homebrew/bin` is not searched. Actual location: `/opt/homebrew/bin/gh` (v2.87.2). |
| Mitigation | Confirmed the binary exists by testing the full path directly (`/opt/homebrew/bin/gh --version`). Ran remote checks using the full path. |
| Prevention | **Template fix required**: `doctor.sh` must probe well-known install locations as fallbacks when `command -v` fails. See Recommendations below. |

## Insights and Learnings

### Technical

- **Claude Code sandbox PATH is minimal**: The sandbox environment only includes `/usr/bin:/bin:/usr/sbin:/sbin`. It does not source the user's shell profile (`.zshrc`, `.bash_profile`). This means any tool installed via Homebrew (`/opt/homebrew/bin`), nvm/fnm (`~/.nvm/`, `~/.local/share/fnm/`), cargo (`~/.cargo/bin`), or other user-space package managers will not be found by `command -v`.
- **`command -v` is necessary but not sufficient**: It's the right first check, but a diagnostic script that runs in restricted environments must also probe well-known installation paths as fallbacks.
- **The npx shim depends on node**: Even when `npx` is found by absolute path, it fails if `node` isn't on PATH. The MCP npx path check (`[ -x "$mcp_npx_path" ] || command -v "$mcp_npx_path"`) tests executability but not runnability.

### Process

- **Diagnostic tools must be reliable in all execution contexts**: `doctor.sh` is invoked two ways — directly from a terminal (`bash scripts/doctor.sh`) and via Claude Code (`/doctor`). The script works correctly in the first context but produces false results in the second. Since `/doctor` is the primary way participants will run it during the workshop, this is a high-impact bug.
- **False positives are worse than false negatives for trust**: A diagnostic that says "everything is fine" when something is broken causes one debugging session. A diagnostic that says "5 things are broken" when nothing is broken causes the user to stop trusting the diagnostic entirely. The second failure mode is harder to recover from.
- **Dry runs catch environmental assumptions**: This is the fourth dry run and the first to surface this specific issue. Earlier dry runs focused on bootstrap flow and feature development. This session's focus on the diagnostic itself revealed the environmental assumption baked into every CLI check.

### Recommendations for `doctor.sh`

1. **Add a `resolve_cmd` helper function** that checks `command -v` first, then probes a list of well-known paths:

   ```bash
   resolve_cmd() {
       local cmd="$1"
       # Try PATH first
       if command -v "$cmd" &>/dev/null; then
           command -v "$cmd"
           return 0
       fi
       # Fallback: probe well-known locations
       local paths=(
           "/opt/homebrew/bin/$cmd"       # macOS Homebrew (Apple Silicon)
           "/usr/local/bin/$cmd"           # macOS Homebrew (Intel) / manual installs
           "$HOME/.cargo/bin/$cmd"         # Rust/cargo installs
           "$HOME/.local/bin/$cmd"         # pipx, uv, etc.
       )
       # Tool-specific paths
       case "$cmd" in
           claude) paths+=("$HOME/.claude/local/$cmd") ;;
           npx|node|npm) paths+=(
               "$HOME/.nvm/versions/node/$(ls "$HOME/.nvm/versions/node/" 2>/dev/null | tail -1)/bin/$cmd"
               "$HOME/.local/share/fnm/aliases/default/bin/$cmd"
           ) ;;
       esac
       for p in "${paths[@]}"; do
           if [ -x "$p" ]; then
               echo "$p"
               return 0
           fi
       done
       return 1
   }
   ```

2. **Use `resolve_cmd` for every CLI check** instead of raw `command -v`. When a tool is found via fallback path, report PASS with the resolved path (not WARN about PATH):

   ```bash
   if resolved=$(resolve_cmd gh); then
       pass "CLI Tools" "gh found at $resolved ($(\"$resolved\" --version | head -1))"
   else
       fail "CLI Tools" "gh not found" "brew install gh"
   fi
   ```

3. **Add a "restricted PATH" detection banner**: If the script detects it's running in a restricted environment (e.g., `PATH` lacks `/opt/homebrew/bin` on macOS), print a one-line notice at the top so the user understands why fallback path resolution is being used.

4. **Fix the MCP npx path check**: Instead of just testing `[ -x "$mcp_npx_path" ]`, resolve the actual `npx` binary and check that `node` is also resolvable. If `npx` is found but `node` isn't, the MCP server won't start — that's a real problem worth reporting.

5. **Consider moving remote checks into the script**: Currently, remote checks live in the `/doctor` slash command instructions and depend on `gh` being available to the Claude Code agent. If `doctor.sh` performed these checks itself (when `gh` is available), the slash command would just parse the output — no duplication, no PATH dependency in the slash command layer.
