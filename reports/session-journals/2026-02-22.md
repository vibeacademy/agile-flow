## Challenges and Mitigations

### 1. Dead hero button — `#chat` anchor with no matching element

| Field | Value |
|-------|-------|
| Challenge | The "Ask AI" button above the fold did nothing when clicked |
| Root cause | **Agile-flow template issue**: The hero component was scaffolded with `<a href="#chat">Ask AI</a>`, but the ChatPanel component was built as a slide-out with internal state — no element with `id="chat"` was ever created. The "Check Fit" button worked because `FitCheckSection` has `id="fit-check"`. This is a pattern mismatch in the scaffolded code. |
| Mitigation | Replaced anchor with a button that dispatches a custom DOM event (`open-chat`). ChatPanel listens for the event. Lightweight, no state lifting needed. |
| Prevention | **Template improvement**: When scaffolding hero CTAs that reference slide-out panels (not anchor-scrollable sections), the template should wire them with events or callbacks instead of `#hash` anchors. The "Check Fit" anchor pattern doesn't generalize to slide-out UIs. |

### 2. Raw markdown in chat messages

| Field | Value |
|-------|-------|
| Challenge | AI responses showed `**bold**`, `- lists`, etc. as plain text — hard to read |
| Root cause | **Agile-flow template scaffolding gap**: The `ChatMessage` component rendered `{content}` as raw text. No markdown library was included. The system prompt doesn't restrict markdown output, so Claude naturally returns formatted responses. |
| Mitigation | Added `react-markdown` and scoped `.prose-chat` CSS styles. Assistant messages render through ReactMarkdown; user messages stay plain text. |
| Prevention | **Template improvement**: If the template scaffolds a chat UI that consumes LLM output, it should include a markdown renderer from the start. Every major LLM returns markdown by default — rendering it as plain text is always a bug. Consider adding `react-markdown` to the base `package.json` when the chat feature flag is enabled. |

### 3. Duplicate React keys for multi-role companies

| Field | Value |
|-------|-------|
| Challenge | React key warnings when rendering experience cards — Thrivent appears 3 times, NBC Sports Next appears 2 times |
| Root cause | `key={exp.company}` assumed one role per company. The placeholder data had unique companies, but real resume data has multiple roles at the same company. |
| Mitigation | Changed to composite key `key={${exp.company}-${exp.role}}`. Updated test to use `getAllByText` grouped by unique companies. |
| Prevention | Template should use a more unique key from the start (index-based or id field). Single-company assumption is fragile. |

## Insights and Learnings

### Technical

- **Custom DOM events for cross-component communication**: `window.dispatchEvent(new CustomEvent("open-chat"))` is a lightweight pattern for bridging components that don't share React state. Avoids context providers or state lifting for one-way signals. Test with `act()` wrapper for React state updates.
- **`react-markdown` is the right default for LLM chat UIs**: Every LLM returns markdown. Rendering it as plain text is always a bug. Scoped CSS (`.prose-chat`) is cleaner than pulling in `@tailwindcss/typography` for a single component.
- **Resume data is just facts-in-the-prompt**: The entire `resumeData` object is `JSON.stringify`'d and injected into system prompts. No RAG, no vector search. This means every word in `resume-context.ts` directly shapes AI responses — it's both the database and the prompt engineering layer.
- **`resume-context.ts` is safely hand-editable**: It's a TypeScript data file with typed interfaces. `tsc --noEmit` catches structural mistakes. No need for an admin UI.

### Process

- **Two-phase content approach works well**: Phase 1 (baseline from resume PDF) gets structure right. Phase 2 (interactive questionnaire) adds depth and first-person narrative. Trying to do both at once would have been overwhelming.
- **Quick Fix Protocol reduces ceremony overhead**: 5 of today's 7 PRs used the quick fix protocol (no ticket, still branch + PR). This kept velocity high for content and bug fix work that didn't warrant full ticket ceremony.
- **Content work dominates early sessions**: Most of today's effort was content — writing, refining, and validating resume data. The code changes were straightforward. This is expected for a content-driven site, but the agile-flow template's ticket structure is optimized for feature work, not content authoring.

### Process — Agile-Flow Template Observations

- **Template assumes features, not content**: The ticket workflow (issue -> Ready -> In Progress -> In Review -> Done) works well for feature development but is awkward for content authoring tasks. Content PRs (#35, #36, #37) didn't have formal tickets — they used quick fix protocol. The template could benefit from a lightweight "content task" workflow.
- **Scaffolded code needs integration testing assumptions**: The hero button bug (#38) and raw markdown bug (#39) were both cases where scaffolded components worked in isolation but failed when integrated. The template's test coverage focused on unit tests for individual components, missing the integration gap between "Ask AI" button and ChatPanel, and between LLM output format and message rendering.
- **No formal content population phase in template workflow**: The template has phases for architecture, features, and polish — but no explicit phase for "populate with real data." This caused confusion about when to create tickets vs. use quick fix protocol for content work.
