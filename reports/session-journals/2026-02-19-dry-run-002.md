## Friction Points

### F1. CI Workflow References Non-Existent Scripts (Severity: HIGH)

**What happened:** After merging PR #22 (database schema), all 4 CI jobs
failed. The template CI workflow (`.github/workflows/ci.yml`) referenced
shell scripts from the Python template that no longer existed after the
Next.js stack transition:

- `tests/validate-commands.sh`
- `tests/validate-docs.sh`
- `tests/validate-imports.sh`
- `tests/validate-structure.sh`

**Impact:** CI completely broken. No PR could pass checks. Required
rewriting the entire CI workflow mid-session.

**Fix applied:** Rewrote `ci.yml` with Node.js CI jobs (lint, typecheck,
test, build) + kept agent policy linting.

**Template fix needed:** `bootstrap-architecture` Phase 2 must rewrite
`ci.yml` when it transitions the stack from Python to Node.js (or
whatever the target stack is). The template ships with Python-specific CI
that becomes invalid after stack transition.

---

### F2. Markdown Linter Scans node_modules (Severity: MEDIUM)

**What happened:** After fixing F1, the lint CI job still failed. The
markdownlint step was scanning `node_modules/` and found hundreds of
errors in third-party package README files.

**Fix applied:** Added `!node_modules` exclusion to the markdownlint glob
in `ci.yml`.

**Template fix needed:** The CI workflow's markdownlint config should
include `!node_modules` by default.

---

### F3. Preview Deploy Injects Wrong Env Var Names (Severity: HIGH)

**What happened:** The template `preview-deploy.yml` injects generic
Supabase env var names (`SUPABASE_URL`, `SUPABASE_KEY`,
`SUPABASE_SERVICE_KEY`) but the Next.js app expects
`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, and
`SUPABASE_SERVICE_ROLE_KEY`.

**Impact:** Render preview deployments always 502 because the app never
receives its Supabase credentials.

**Fix applied:** Rewrote the env var injection step to use the correct
Next.js variable names.

**Template fix needed:** The `preview-deploy.yml` must be aware of the
app's actual env var names. Either:

- (a) `bootstrap-architecture` generates a stack-specific
  `preview-deploy.yml` during Phase 2, or
- (b) the workflow reads env var names from `render.yaml` dynamically, or
- (c) the template uses a mapping config file that both `render.yaml` and
  the workflow reference.

Option (a) is the most straightforward for the workshop.

---

### F4. Preview Deploy Missing Auth Redirect Configuration (Severity: HIGH)

**What happened:** The template `preview-deploy.yml` had no step to add
the Render preview URL to the Supabase branch's auth redirect URL
allowlist. Without this, any auth flow (login, register) on the preview
would fail because Supabase rejects the callback URL.

**Impact:** Preview environments unusable for testing auth flows.

**Fix applied:** Added a "Configure Supabase branch auth redirect URLs"
step that PATCHes the branch's auth config via the Supabase Management
API, adding `https://kitchenista-pr-{N}.onrender.com/**` to
`uri_allow_list` and setting `site_url`.

**Template fix needed:** Port this step into the template's
`preview-deploy.yml`. Use a variable for the service name pattern instead
of hardcoding.

---

### F5. Preview Deploy Has No CI Gate (Severity: MEDIUM)

**What happened:** The template `preview-deploy.yml` did not depend on CI
passing. Broken code (lint failures, test failures) would still trigger a
preview deployment, wasting Render build minutes and creating confusing
502s.

**Fix applied:** Added `ci` job dependency: `needs: [ci, wait-for-supabase]`
with `if: always() && needs.ci.result == 'success'`.

**Template fix needed:** Add CI as a prerequisite for the deploy-preview
job.

---

### F6. Preview Deploy Doesn't Recover From Initial Build Failure (Severity: HIGH)

**What happened:** Render auto-creates a preview service when a PR is
opened, but that first build has no Supabase env vars (they aren't set on
the production Render service either). The build fails or the app crashes
with a 502. The template workflow saw `build_failed` status and set
`preview_ready=false`, so it never injected env vars or triggered a
redeploy.

**Impact:** Preview environments permanently stuck at 502. The workflow
needed to inject env vars and redeploy *regardless* of the initial deploy
status.

**Fix applied:** Changed the workflow to set `preview_found=true` whenever
the Render preview service exists (any deploy status), then always inject
env vars and trigger a redeploy.

**Template fix needed:** The workflow should treat "service found" and
"service ready" as separate concerns. Env var injection should happen as
soon as the service is found, not only when the first deploy succeeds.

---

### F7. Preview Deploy Doesn't Smoke Test (Severity: LOW)

**What happened:** The template workflow had no health check after
redeploying. There was no feedback about whether the preview actually
came up healthy.

**Fix applied:** Added an inline smoke test step that polls
`/api/health` with retries.

**Template fix needed:** Add a smoke test step. The leadflo project uses
a `scripts/smoke-test.sh` but for simplicity an inline curl loop is
sufficient for the workshop template.

---

### F8. Production Render Service Has No Env Vars (Severity: HIGH)

**What happened:** After deploying the app to Render via `render.yaml`,
the production service had zero environment variables configured. The
`render.yaml` declares `sync: false` for all Supabase vars (correct —
secrets shouldn't be in the blueprint) but nobody set them in the Render
dashboard.

**Impact:** Production app 502s. Participants would see a broken deploy
immediately after bootstrap.

**Template fix needed:** The `bootstrap-architecture` or
`bootstrap-workflow` phase should explicitly prompt the participant to set
env vars on Render. Better yet, the `FIRST-DAY.md` walkthrough should
include a "Configure Render Environment Variables" step with a checklist:

- [ ] `NEXT_PUBLIC_SUPABASE_URL`
- [ ] `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- [ ] `SUPABASE_SERVICE_ROLE_KEY`

And the `PRE-WORK-CHECKLIST.md` should include creating a Render API key
and noting the Render service ID (for the preview-deploy workflow).

---

### F9. PR Comment Duplicates on Re-Push (Severity: LOW)

**What happened:** The template `preview-deploy.yml` used
`createComment` without checking for existing comments. Every push to a
PR branch created a new "Preview Deployment" comment.

**Fix applied:** Added logic to find and update existing comments
(search for `## Preview Deployment` prefix).

**Template fix needed:** Use the find-and-update pattern in the template.

---

### F10. shadcn/ui Not Pre-Installed (Severity: LOW)

**What happened:** Ticket #9 (user registration) required shadcn/ui
components (Button, Input, Label, Card) but shadcn/ui was not initialized
during `bootstrap-architecture`. The agent had to run `npx shadcn@latest init`
and add components mid-ticket.

**Impact:** Extra time spent on setup during a feature ticket. The init
process modifies `globals.css`, `tailwind.config.ts`, and adds
`components.json` — a large diff that clutters the feature PR.

**Template fix needed:** If the architecture phase selects shadcn/ui as
the component library, `bootstrap-architecture` should initialize it and
install base components (button, input, label, card) as part of the
scaffold. This keeps feature PRs focused on feature code.

---

### F11. Epic Issues Created With Old Format (Severity: LOW)

**What happened:** The `bootstrap-workflow` Phase 4 created 6 epic
issues (#1-#6) using a simple format (title + acceptance criteria) rather
than the Agentic PRD Lite format with 4 Power Sections. The
implementation tickets (#7-#20) were correctly formatted.

**Impact:** Epics are not directly worked by agents (they're parent
containers), so the format mismatch doesn't block anything. But it's
inconsistent and could confuse participants who expect uniform formatting.

**Template fix needed:** Either:

- (a) Document that epics use a lighter format (intentional), or
- (b) Give epics a distinct template in `bootstrap-workflow` with
  sections like: Overview, Child Tickets, Acceptance Criteria.

Option (a) is simpler and probably correct — epics are decomposed into
child tickets that have the full format.

---

### F12. Missing GitHub Secrets Documentation (Severity: MEDIUM)

**What happened:** The preview-deploy workflow requires 4 GitHub secrets
(`SUPABASE_ACCESS_TOKEN`, `SUPABASE_PROJECT_REF`, `RENDER_API_KEY`,
`RENDER_SERVICE_ID`) plus 3 fallback secrets for production Supabase
credentials. None of these were documented in the getting-started guide
or pre-work checklist.

**Impact:** Workflow silently fails or skips steps. Participants won't
know what secrets to configure or where to find the values.

**Template fix needed:** Add a "Configure GitHub Secrets" section to
`FIRST-DAY.md` and `PRE-WORK-CHECKLIST.md` with a table of required
secrets, where to find each value, and which features they unlock.

| Secret | Source | Required For |
|--------|--------|-------------|
| `SUPABASE_ACCESS_TOKEN` | Supabase Dashboard → Account → Access Tokens | Preview branch credentials |
| `SUPABASE_PROJECT_REF` | Supabase project URL (the ref segment) | Preview branch detection |
| `RENDER_API_KEY` | Render Dashboard → Account Settings → API Keys | Preview env var injection |
| `RENDER_SERVICE_ID` | Render service URL (starts with `srv-`) | Preview service discovery |
| `NEXT_PUBLIC_SUPABASE_URL` | Supabase project settings | Production fallback |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Supabase project settings | Production fallback |
| `SUPABASE_SERVICE_ROLE_KEY` | Supabase project settings | Production fallback |

## Observations

### What Worked Well

- **4 Power Section ticket format** — When tickets had all 4 sections
  (Environment Context, Guardrails, Happy Path, Definition of Done), the
  agent produced correct, well-scoped implementations on the first try.
  Tickets #7-#10 all landed cleanly.

- **Three-account workflow** — Separation of worker/reviewer/human worked
  as designed. The va-worker created PRs, va-reviewer posted GO reviews,
  human merged. No role confusion.

- **Agent self-correction on guardrails** — When implementing RLS
  policies (#8), the agent initially created a DELETE policy on menus,
  then caught the guardrail violation ("Do NOT grant DELETE on orders or
  menus to any role via RLS") and removed it before pushing.

- **Test coverage** — All 4 tickets produced meaningful tests. 52 tests
  total, all passing. Tests caught real issues (mocking patterns,
  middleware behavior).

### What Needs Improvement

- **Preview environments are the #1 gap.** The entire Supabase + Render
  preview pipeline was broken out of the box. A workshop participant would
  hit a 502 on their first PR and have no idea why. This must be
  seamless.

- **Env var configuration is undocumented.** Neither the Render production
  service nor the GitHub secrets are mentioned in any setup guide.
  Participants need a clear checklist.

- **Stack transition is messy.** `bootstrap-architecture` scaffolds
  Next.js but leaves behind Python CI artifacts. The transition should be
  atomic — update CI, cleanup old files, initialize tooling (shadcn, etc.)
  in one coherent step.
