## Challenges and Mitigations

### 1. `hashFiles()` in GitHub Actions job-level `if`

| Field | Value |
|-------|-------|
| Challenge | CI workflow failed with "workflow file issue" when using `hashFiles('pyproject.toml')` in a job-level `if` condition |
| Root cause | `hashFiles()` at job level runs before `actions/checkout`, so the workspace is empty and the function always returns an empty string |
| Mitigation | Moved the check to a step-level bash `[ -f "pyproject.toml" ]` after checkout, using output variables to gate subsequent steps |
| Prevention | Never use `hashFiles()` in job-level `if` — only use it in step-level conditions after checkout |

### 2. Auto-fix workflow pushing commits to PR branch

| Field | Value |
|-------|-------|
| Challenge | `git push` rejected because the auto-fix workflow had pushed a formatting commit to the PR branch |
| Root cause | The newly created `auto-fix.yml` workflow actually worked and reformatted imports in `app/main.py`, pushing a commit. Local push then diverged. |
| Mitigation | Pulled remote changes first, then pushed. This is expected behavior — the auto-fix workflow is working as designed. |
| Prevention | Always `git pull` before pushing to a PR branch when auto-fix is enabled |

### 3. CI checks not attaching to PR (PR #53)

| Field | Value |
|-------|-------|
| Challenge | CI checks ran and passed but didn't attach to the PR as required status checks. `gh pr checks` showed no checks. PR was blocked by rulesets requiring 5 status checks. |
| Root cause | Likely a timing issue or GitHub race condition between the CI run and the PR creation |
| Mitigation | Pushed an empty commit to retrigger CI. Checks attached properly on the second run. |
| Prevention | If checks don't attach after PR creation, an empty commit retrigger is the standard fix |

### 4. Shellcheck SC2129 and markdownlint MD029

| Field | Value |
|-------|-------|
| Challenge | `bootstrap.sh` failed shellcheck (sequential `>>` redirections), `FAQ.md` failed markdownlint (ordered list numbering) |
| Root cause | SC2129: three `echo >> file` lines should use grouped redirection `{ } >> file`. MD029: list item started at `4.` instead of `1.` (markdownlint expects `1/1/1` style) |
| Mitigation | Quick fix commit — grouped the redirections, changed `4.` to plain text |
| Prevention | Run `shellcheck` and `markdownlint` locally before pushing |

## Insights and Learnings

### Technical

- **GitHub Actions `hashFiles()` scope**: Only reliable in step-level `if` conditions after checkout. At job level, the workspace is empty. This applies to any function that reads files.
- **Auto-fix workflows create merge hazards**: When a workflow pushes to the PR branch, local work must `git pull` before continuing. This is a feature, not a bug, but requires awareness.
- **GitHub rulesets vs branch protection**: This repo uses rulesets (not classic branch protection). `gh pr merge --admin` does not bypass rulesets — only repo admins through the UI can.
- **Shellcheck SC2129**: Multiple sequential `>> file` redirects should be grouped. This is a style warning but the CI linter treats it as an error.
- **Context window efficiency**: Front-loading critical rules in CLAUDE.md and using `NON-NEGOTIABLE PROTOCOL` blocks in agent definitions measurably improves agent compliance. The "lost-in-the-middle" problem is real.

### Process

- **Batching related issues into single PRs** dramatically reduces review overhead and merge ceremony. 11 PRs for 41 issues (average 3.7 issues per PR) was efficient.
- **The populate-ready → implement-all → create-PR → merge → move-to-done cycle** is a reliable cadence for burning down a project board with an AI agent.
- **Net deletion is a feature**: PR #53 removed 2,041 lines (marketing agent, marketing commands, redundant devops agents). Templates should ship lean.

### Domain

- **"Agentic PRD Lite" insight**: The session ended with a strategic realization that ticket authoring is the bottleneck for agent productivity. Agents need dense, constraint-rich specs (environment context, guardrails, happy-path flow, definition of done) — not traditional user stories. The backlog groomer agent needs to be updated to scope tickets for agent consumption, not human consumption.
- **Product-level vs feature-level docs serve different purposes**: The product PRD and architecture docs establish the "world." Feature-level tickets establish the "mission." Both are needed but they have different structures.
